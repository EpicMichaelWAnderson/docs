1. Why transitions exist at all

Modern GPUs don't just execute draw calls serially against a single coherent memory view. They have:

  • Pipelined hardware — vertex, pixel, compute, raster, ROP, and copy units all run in flight at once. One stage writing while another reads the same memory is a hazard unless you tell the GPU to wait.
  • Layered caches — L1, L2, ROP caches, texture caches. Data written through one path isn't automatically visible through another until caches are flushed/invalidated.
  • Compressed texture formats — a GPU may store a render target in an internal compressed form while it's an RTV, but must decompress it before shader reads. The driver needs a signal for when to switch.
  • Multiple queues — graphics and async compute can run on separate hardware engines with separate caches and separate command submissions. A resource written on one queue needs explicit synchronisation before it's safe to read on another.

The driver can't infer these requirements from command-buffer content alone. Someone has to tell it: "this resource was a render target, now it needs to be an SRV — insert the right caches flush / decompress / queue wait." That's what a barrier is. What UE5 calls a transition is its platform-agnostic description of such a barrier.

The RHI's job: accept a high-level "before state → after state" declaration from the renderer, and emit whatever sequence of native barriers / fences / cache ops the platform requires to make it correct.

2. The RHI transition model — declarations, not barriers

An FRHITransition is not a barrier. It's a declaration of a state change that the RHI will translate into whatever native barrier sequence the platform needs.

The three things a transition describes

  • What's changing — one or more FRHITransitionInfo entries, each naming a resource, an "access before" and "access after", and optionally a subresource range.
  • Between which pipes — source and destination ERHIPipeline flags (graphics, async compute, or both). This tells the RHI whether a cross-queue fence is involved.
  • Creation flags — optional hints like NoFence, AllowDecayPipelines, MaintainCompression that modify the RHI's choices.
Where to look: FRHITransitionInfo, FRHITransitionCreateInfo, and FRHITransition live in Engine/Source/Runtime/RHI/Public/RHITransition.h. Access flags are in RHIAccess.h. Pipeline flags are in RHIPipeline.h.

3. Access modes — ERHIAccess

A bitmask that names every way a resource can be read or written. The transition's "before" and "after" are both ERHIAccess values — the backend translates the pair into the right native barrier.

CategoryExample modesWhat it means
Read-only SRVCompute, SRVGraphicsPixel, SRVGraphicsNonPixel, VertexOrIndexBuffer, IndirectArgs, CopySrc, Present, DSVRead, BVHRead Resource is being read by one or more stages. Multiple read modes can coexist.
Read-write UAVCompute, UAVGraphics, RTV, CopyDest, DSVWrite, BVHWrite Resource is being written. Usually exclusive — only one writer at a time.
Transient Discard Contents are undefined / no access required. Used by transient allocator (see §9).

"From SRV to UAV" is the common pattern for a texture about to be written by a compute pass; "from RTV to SRV" is the pattern for a render target that a later pass will sample; "from Present to RTV" is the start-of-frame transition on the back buffer.

Read-mode bits are additive. A resource in SRVCompute | SRVGraphicsPixel means "readable from compute shaders and pixel shaders simultaneously." Write modes are typically exclusive. When you see a transition that looks like "from (SRV | UAV) to (SRV)", it's saying "we were writable and readable, now we're read-only" — the RHI knows to flush the UAV caches.

4. Split-phase lifecycle — begin, work, end

Transitions are issued as two separate commands around the work that depends on the new state.

This two-phase model is the mechanism for split barriers, one of the main performance wins of modern graphics APIs. The idea:

  • Begin tells the GPU "this resource is going to transition — start flushing/invalidating whatever you need." The hardware can overlap that work with unrelated commands that come after Begin.
  • End tells the GPU "the transition must be complete now, because the next commands depend on it." If the GPU has already finished the flush, End is cheap.

Putting unrelated work between Begin and End lets the barrier's latency hide behind real work. Squashing them together (Begin immediately followed by End) falls back to a regular non-split barrier.

Where to look: the two commands are FRHICommandBeginTransitions and FRHICommandEndTransitions in RHICommandList.h. Their Execute() paths call the backend's RHIBeginTransitions / RHIEndTransitions, which ultimately emit native barriers. Cleanup bookkeeping uses atomic state on FRHITransition (MarkBegin/MarkEnd in RHITransition.h).

What goes in Begin vs End, and where they belong

Begin and End both receive the same transition object. You call RHICreateTransition once to describe the state change, then hand that pointer to both commands. Begin is "start the transition"; End is "the transition must be complete now." Neither is a half-description — the declaration lives in the transition object itself.

Placement rule 1

Begin goes after the last OLD-state use

Emit it as early as you know the resource won't be used in the before-state again. Earlier Begin = more room for the barrier's latency to overlap with other work.

Placement rule 2

End goes before the first NEW-state use

This one is strict. If End comes after the first new-state use, that use sees stale data, wrong layout, or uncoalesced caches — the classic barrier bug.

The gap between Begin and End is where the win lives. Anything that doesn't touch the transitioning resource is fair game — unrelated draws, dispatches, copies, other transitions. The larger the gap, the more of the barrier's flush/decompress/layout-change cost the GPU can hide behind real work. If Begin and End land adjacent (no unrelated work between them), most backends collapse them into a single full barrier — no worse than no split, but no better either.
In practice you rarely place these yourself. RDG sees the whole frame's pass graph, so it can place Begin right after the last writer, End right before the first reader, and use the passes in between as the hiding gap. Hand-placing Begin/End correctly across a frame is fiddly — let RDG do it unless you have a specific reason not to.

5. Per-subresource granularity

A transition doesn't have to move an entire resource — it can target a specific mip, array slice, or aspect (depth vs stencil).

This matters for pipelines like cascaded shadow maps (different mips rendered in different passes), mip generation (cascading writes-then-reads), or when you're reading from mip N while writing to mip N+1.

The default is "whole resource" — a convenience value (kAllSubresources) means every subresource at once. Use it when there's no advantage to being finer; use explicit ranges when different subresources are in genuinely different states at the same time.

6. Cross-pipe transitions — graphics ↔ async compute

A transition can span different pipelines, not just different states. "This UAV was written by async compute, now I want to read it from the pixel shader" needs more than a cache flush — it needs cross-queue synchronisation so the graphics queue doesn't race ahead.

wrote UAV"]:::ac F["queue fence /
semaphore / event"]:::fence GFX["graphics:
will sample as SRV"]:::gfx AC --> F --> GFX classDef ac fill:#4a2e15,stroke:#f0883e,color:#e6edf3 classDef fence fill:#3b2a55,stroke:#d2a8ff,color:#e6edf3 classDef gfx fill:#1d3f1f,stroke:#7ee787,color:#e6edf3

When the source and destination ERHIPipeline flags differ, the RHI emits a queue fence (D3D12), semaphore (Vulkan), or platform equivalent so the consumer waits for the producer. The renderer can opt out with the NoFence flag when it's managing synchronisation itself.

State decay: an AllowDecayPipelines flag lets a resource that went "Graphics | AsyncCompute" transition back to "Graphics" without another explicit fence, on the assumption the async work has already been synchronised elsewhere. It's the kind of optimisation RDG emits automatically — you don't normally touch it by hand.

7. How a transition reaches the driver

Each platform backend translates FRHITransition into its own native barrier calls. The RHI-layer record is the same everywhere; only the emission differs.

BackendNative callSource pointer
D3D12 (legacy) ID3D12GraphicsCommandList::ResourceBarrier D3D12RHI/Private/D3D12LegacyBarriers.cpp
D3D12 (enhanced) ID3D12GraphicsCommandList7::Barrier D3D12RHI/Private/D3D12EnhancedBarriers.cpp
Vulkan vkCmdPipelineBarrier2KHR / VkSemaphore, VkEvent VulkanRHI/Private/VulkanBarriers.cpp
Other consoles Platform-specific (NVN, AGC, etc.) Under Engine/Platforms/*/Source/Runtime/

Where state tracking lives

The RHI layer is thin — it records the declared transition but doesn't decide which native barriers are strictly necessary. That decision is per-backend:

  • D3D12 legacy tracks per-subresource state internally and emits barriers only for actual state differences.
  • D3D12 enhanced and Vulkan translate each transition's access flags to stage/access masks on the fly, using the API's built-in access-based barrier semantics.
  • Consoles often have their own mini state-tracker that reflects hardware-specific compression formats and cache behaviour.

8. What the RHI does NOT do for you

The RHI transition system is explicit. It won't silently fix hazards — if you read a resource as an SRV without having transitioned it out of its previous state, you'll hit validation errors (or, worse, platform-specific corruption).

Your job

Explicit declarations

  • Every state change between passes — even if "obvious."
  • Cross-queue sync when two pipes share a resource.
  • Per-subresource ranges when different mips/slices diverge.
  • Wrapping each transition with Begin/End around the work that depends on it.
The RHI's job

Choosing the native barriers

  • Converting access pairs into cache flushes, decompresses, layout changes.
  • Coalescing multiple transition infos in a single Begin/End batch into the fewest native calls.
  • Inserting cross-queue fences when pipelines differ.
  • Applying platform-specific workarounds when state combos need them.
Initial state — a resource has a default access at creation (determined by RHIGetDefaultResourceState). Your first transition must correctly name that state as the "before" — no guessing. Getting it wrong on D3D12 legacy backends is a common source of debug-layer errors.

9. Aliasing barriers — a different beast

When a transient-resource allocator reuses the same physical memory for resource A (now finished) and resource B (about to start), the GPU needs to know the old tile-cache contents belonged to a different resource. That's an aliasing barrier — related to transitions, but structurally separate.

The RHI's aliasing-barrier machinery lives on FRHITransientAliasingInfo (acquire/discard actions) and travels alongside regular transitions through the same Begin/End commands. Conceptually:

  • Access transitions — "same resource, different state."
  • Aliasing barriers — "same memory, different resource."

You'll rarely author these by hand; the transient resource allocator emits them automatically.

10. Who actually writes transitions (hint: not you)

In practice most renderers never call RHIBeginTransitions directly. They describe resources and access per pass in the Render Dependency Graph (RDG), and RDG emits transitions for them.

RDG knows which pass reads what and which writes what, so it can schedule transitions optimally — batching them, splitting Begin/End across unrelated passes for maximum overlap, and coalescing redundant state changes. You can still hand-author transitions at the RHI layer when you need something RDG can't express, but that's the exception.

Where to look: FRDGBarrierBatchBegin::CreateTransition and friends in RenderCore/Public/RenderGraphPass.h are where RDG turns per-pass access declarations into RHI transitions.

12. One sentence

An RHI transition is a platform-agnostic "this resource is moving from state A to state B" declaration, split into Begin/End phases around the work that depends on the new state, that each backend translates into whatever native barriers — cache flushes, layout changes, queue fences — the underlying GPU actually needs.
