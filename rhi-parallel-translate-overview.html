1. The pipeline at 30,000 feet

A frame in UE5 moves through four distinct phases, each on its own thread context. Understanding this split is the foundation for everything else.

produces frame data"]:::gt RT["RenderThread

submits to driver"]:::rhi GPU["GPU"]:::gpu GT -- "ENQUEUE_RENDER_COMMAND" --> RT RT -- "Submit()" --> W W -- "native cmd bufs" --> RHI RHI --> GPU classDef gt fill:#2d3745,stroke:#58a6ff,color:#e6edf3 classDef rt fill:#1b3a5c,stroke:#58a6ff,color:#e6edf3 classDef worker fill:#1d3f1f,stroke:#7ee787,color:#e6edf3 classDef rhi fill:#4a2e15,stroke:#f0883e,color:#e6edf3 classDef gpu fill:#4a1a17,stroke:#ff7b72,color:#e6edf3
  • GameThread runs gameplay, queues work for the renderer via ENQUEUE_RENDER_COMMAND.
  • RenderThread translates gameplay intent into RHI calls. It doesn't touch the GPU directly — it records a linear plan into a command list.
  • Translator workers walk the recorded stream and populate platform-native command buffers (D3D12, Vulkan, Metal, NVN, AGC, …). This is where "parallel translate" lives — nothing is submitted to the driver yet, just recorded into native buffers.
  • RHIThread submits those finished native buffers to the platform driver. Only at submit do driver-visible operations actually begin.
The key split: recording is serial (one RenderThread), execution is parallel (many translator workers). Commands are queued top-of-pipe and executed bottom-of-pipe — the entire RHI vocabulary is built around that distinction.

2. The immediate list — the frame's spine

There's one command list that defines a frame's canonical ordering: FRHICommandListImmediate, a persistent singleton owned by the executor (not recreated per frame — reset and reused). The RenderThread holds a reference to it and records into it directly.

Two kinds of command list

  • The immediate list — one per frame, always single-threaded for legacy reasons. It's the trunk: it holds the canonical ordering, the global fences, and any work the renderer recorded without explicitly asking for parallelism.
  • Non-immediate sub-command-lists — the renderer builds these explicitly (typically inside a parallel render pass) and queues them onto the immediate list with QueueAsyncCommandListSubmit. These are eligible for parallel translation.

So parallel translate is not the executor secretly splitting the immediate list behind the renderer's back. It's the renderer deliberately carving work off into sub-lists that the executor can then translate in parallel — while the immediate-recorded work stays on a single-threaded chain.

Where to look: the class chain is FRHICommandListBase → FRHIComputeCommandList → FRHICommandList → FRHICommandListImmediate — each a strict derivation of the previous, with the immediate subclass flagged via PersistentState.bImmediate. The singleton lives in FRHICommandListExecutor::GRHICommandList.CommandListImmediate, fetched via FRHICommandListImmediate::Get(). Sub-command-lists are queued onto it with QueueAsyncCommandListSubmit.

3. Top-of-pipe vs bottom-of-pipe

This pair of terms shows up everywhere in the RHI source. They name the two halves of the recording/execution split.

Top-of-pipe

When commands are recorded

You're on the RenderThread. The command list is being written into. Nothing has translated or executed yet. RHICmdList.IsTopOfPipe() returns true.

Calls like SetShader, DrawIndexedPrimitive, CreateShaderResourceView append entries to the stream. A lambda passed to EnqueueLambda has been queued, but its body hasn't run.

Bottom-of-pipe

When commands execute

A translator worker has picked up a chunk of the stream and is walking it. Each recorded command's Execute() runs, including any lambda bodies. RHICmdList.IsBottomOfPipe() returns true.

This is where native-API calls actually happen and where fence events actually signal. Also the context in which deferred resource deletion runs.

Rule of thumb: anything that observes or mutates live GPU-side state happens bottom-of-pipe. Anything that describes what the GPU will eventually do happens top-of-pipe. EnqueueLambda is the bridge — recorded top-of-pipe, executed bottom-of-pipe.
Bypass mode: with r.RHICmdBypass=1 there is no deferred pipeline at all — every command executes inline on the calling thread, and IsBottomOfPipe() is true at record time too. Useful for debugging; slower in general.

Why code checks which side it's on

Once you know the vocabulary, the checks you see sprinkled through the RHI source fall into three clear patterns. Knowing which one you're looking at tells you a lot about a method's contract.

PatternWhat the branch means
if (Bypass()) { … } else ALLOC_COMMAND(…) Top-of-pipe entry with a bypass shortcut. In normal mode the method records a command to run later; in bypass mode there's no deferred pipeline, so it runs inline right now. The method is not intended to be called from inside a lambda body.
if (IsBottomOfPipe()) { … } else ALLOC_COMMAND(…) Genuinely dual-purpose. Safe to call from either record time or execution time. If the caller is already inside a lambda, run inline; otherwise record. EnqueueLambda is the archetype.
checkf(IsTopOfPipe() || Bypass(), …) Record-time only. This method describes deferred work and makes no sense to call once deferral has started. Reject the caller loudly. Applied to RHIThreadFence, EnqueueLambdaMultiPipe, buffer Lock/Unlock, etc.
The predicate relationship that ties it together: 
IsBottomOfPipe()  =  Bypass() || IsExecuting()
IsTopOfPipe()     =  !IsBottomOfPipe()
So IsTopOfPipe() is false in two situations: bypass mode, and while a previously-recorded command is actually executing. A method gated by IsTopOfPipe() || Bypass() accepts bypass but rejects mid-execution. A method branching on IsBottomOfPipe() embraces both.

A second reason: lifetime-aware optimisations

Beyond gating what's legal, some methods use the distinction to decide whether captured data needs to be copied into the command list's arena. If execution will be deferred, caller-stack memory won't survive; if the lambda will run inline, it will. A typical shape:

TConstArrayView<FooResource> Captured =
    RHICmdList.IsTopOfPipe()
        ? RHICmdList.AllocArray(Input)   // deferred: copy into arena that outlives this call
        : Input;                         // inline: caller's stack is still alive

RHICmdList.EnqueueLambdaMultiPipe(..., [Captured](...) { ... });
So the mental rule: an IsTopOfPipe() branch in an otherwise dual-mode method usually isn't about legality — it's about lifetime. "Will this capture still be valid when the lambda runs? If execution is deferred, no — copy it into the arena. If execution is immediate, yes — pass it through."

4. Where the stream gets chopped

At submit time the executor chops the queued command lists into translator-task chunks. A few simple questions decide whether the next list appends to the current chunk or starts a new one.

Safety

Can this list translate in parallel at all?

Some lists are flagged as needing single-threaded translation — anything that will mutate shared RHI state, the immediate list itself, plus a few platform-specific opt-outs. Those run alone; the chain splits on either side.

Size

Has this chunk grown too large?

A command-count ceiling keeps any single translator task from becoming a stall point. Once a chunk fills up, the next list opens a fresh one.

Mode

Does the next list change the translation mode?

Parallel chains and single-threaded chains don't mix. When the next list's mode differs from the current chain's, a split is forced.

Structure

Are we crossing a render-pass boundary?

Parallel render passes translate as child tasks of a parent list. Entering or leaving that parent/child relationship always starts a new chain so the task graph can wire up the right dependencies.

One implication worth remembering: recording an RHIThreadFence(true) anywhere on a list marks that list as "mutates shared state" and demotes the whole list to single-threaded translation — not just the commands around the fence. For narrow mutations, put the fenced work on a small dedicated sub-list so the rest of the frame stays parallel.
Where to look: the veto is AllowParallelTranslate(). The split predicates live in the executor's submit path. CVars under r.RHICmd.ParallelTranslate.* control the master switch, chunk size ceiling, and a few relaxation knobs; GRHISupportsParallelRHIExecute is the platform gate.

5. Shared state — the rare hazard

Most RHI commands only touch local state: the worker's own native command buffer, its own scratch allocator. Two workers translating two different chunks don't look at each other's memory — so they can't race. Parallel-safety is the default, not something fences manufacture.

A few commands do touch shared state — inserting into a resource-view linked list, freeing a bindless handle, walking a rename list during defrag. If chunk A mutates and chunk B walks, concurrent translation races. Two clean fixes exist.

Fix A

Lock it

Take a critical section around every mutate and every walk. Any chunk can touch shared state safely; contention is handled at runtime.

Example: D3D12's view rename-listener list with FCriticalSection.

Fix B

Sequence it

Force the mutator chunk to finish before any later chunk starts. No lock needed — the schedule itself keeps them apart.

Example: NVN and AGC view linked lists, guarded by EnqueueLambda + RHIThreadFence(true).

6. Fences: happens-before in the task graph

A fence is not a gate the RHI thread polls. It's a happens-before edge injected into the task graph.

  1. At record time, the fence creates a signal token (a graph event) and drops a marker into the command stream.
  2. At submit time, every chunk enqueued after the marker is born with a prerequisite on that token — before it even starts.
  3. When the chunk containing the marker runs, it flips the token from pending to signalled. Only then does the scheduler release the dependent chunks.
Nothing scheduled after the fence exists in the scheduler's eyes until the signal fires. That's the whole mechanism — no locks, no polling.
Where to look: FRHICommandListBase::RHIThreadFence(bool bSetLockFence) allocates an FGraphEventRef, wraps it in a command, and (if bSetLockFence is true) stashes it so every future translate task takes it as a prerequisite. The command's Execute() calls DispatchSubsequents() to signal.

7. Fences are one-sided joins

This is the nuance that trips people up. A fence does not gather all prior work like a fork-join barrier. It only waits on the specific chunk containing the fence marker.

  • Pre-fence chunks keep running concurrently. The fence doesn't collect them. A and B may still be in flight long after C signals.
  • Fence chunk runs normally and signals the token when it reaches the marker.
  • Post-fence chunks are the only ones gated. They inherit the prereq at enqueue time and can't start until the signal fires.
If you need a true barrier — "all prior work done, then continue" — a fence isn't the tool. Use a submit boundary or full flush.

8. When to add a fence

Whenever a lambda writes state that later recorded commands will read, and the write and the read could end up in different translator chunks.

Add a fence

Yes — use RHIThreadFence(true)

  • Creating a resource view that later draws will bind — the view needs to appear in the resource's view list before any rename walk runs.
  • Allocating a shared descriptor slot (bindless, sampler table) that subsequent bindings will index.
  • Registering a transient-resource alias that later commands need to see.
  • Deferred init of persistent state — building an acceleration structure whose pointer is cached and used by later dispatches.
  • Any lambda that mutates RHI-side data another chunk will later read.
Skip the fence

No — it would serialise for nothing

  • Pure draw / bind / dispatch sequences — only touch the worker's own native command buffer.
  • Lambdas whose effects are local (scratch cleanup, logging, per-chunk bookkeeping).
  • Setting platform-specific context state scoped to the current command buffer.
  • Resource creation where no command this frame reads the result — the next submit's boundary syncs you.
Decision test: "could the read-side of my state land in a sibling chunk that the scheduler is free to start concurrently?" If yes, fence. If the read is ordered by GPU command-buffer semantics or by a submit boundary, no fence needed.
Over-fencing is a real cost. Every RHIThreadFence(true) serialises future translators and demotes its own list to single-threaded translation (see §4). Sprinkling them "just in case" collapses the parallelism you enabled. Reach for a lock when contention is rare and narrow.

9. EnqueueLambda and EnqueueLambdaMultiPipe

These are the two ways to inject arbitrary code into the RHI pipeline.

EnqueueLambda — the common case

Records a lambda into the stream at top-of-pipe; the lambda body runs at bottom-of-pipe on whichever translator owns the chunk. The lambda receives the command list and can access its active context.

It doesn't hardcode a pipeline. The command list tracks which pipeline is active at any moment (Graphics or AsyncCompute) via SwitchPipeline, typically inside an RAII scope. A lambda runs with whatever pipeline was active at record time — usually Graphics, sometimes AsyncCompute inside an async-compute scope. Inside the lambda, GetContext() asserts exactly one pipeline is active.

EnqueueLambdaMultiPipe — crossing the graphics / async-compute divide

Same idea, but activates both pipelines simultaneously for the duration of the command. The lambda receives an FRHIContextArray indexed by pipeline, so you can issue work against both contexts in one recorded operation.

EnqueueLambda Exactly one pipeline must be active at execution time — whichever the stream is on. Injecting work into graphics or async-compute (but not both) — the 95% case.
EnqueueLambdaMultiPipe Can activate both pipelines for the command's duration. Recording a single operation that must touch both pipes — cross-pipe transitions, bindless registration visible to both queues, per-pipe view invalidation.
MultiPipe has an optional built-in fence. Passing EThreadFence::Enabled auto-appends a RHIThreadFence(true) — the common case, because MultiPipe lambdas usually mutate shared state both pipes will observe.
Not to be confused with MGPU. "MultiPipe" means multiple queues on one GPU (graphics + async compute). Multi-GPU (alternate-frame rendering, linked adapters) is a separate mechanism layered on top — FRHIGPUMask, linked-adapter objects, etc.

11. One sentence

The RenderThread records a frame-wide plan into one immediate command list; the executor chops that plan (plus any explicitly queued sub-lists) into chunks that translator workers convert to native API calls concurrently; fences are the task-graph edges that retroactively introduce ordering when a lambda needs to mutate shared state without racing what comes next.
An introduction, not a reference. Specifics — flag names, CVar defaults, platform caveats — live in the RHI source under Engine/Source/Runtime/RHI/ and per-platform backends under Engine/Platforms/*/Source/Runtime/.
